## Required Answers

1. Can VCP Bridge provide all six fields?

Yes, but only if the bridge runtime has a trusted session/config layer. A future
bridge can provide `agentAlias`, `agentId`, `requestSource`, `projectId`,
`workspaceId`, and `clientId` as bridge-owned context. Without that layer, the
bridge is not strict-candidate-capable.

2. Do these fields come from trusted bridge runtime context or tool payload?

They must come from trusted bridge runtime context, static bridge config, or a
bridge-generated canonical context object. They must not come from user prompt
text, public MCP tool arguments, or VCP tool payload fields. Tool payload scope
may describe the memory record, but it must not authorize the write that carries
that payload.

3. Which fields can be injected by VCP plugin config?

VCP plugin config can inject or constrain:

- `agentAlias`, as a fixed bridge actor alias or allowlisted alias family
- `requestSource`, as a fixed canonical value such as `vcp-bridge`
- `projectId`, `workspaceId`, and `clientId`, only from operator-controlled
  static config or an allowlisted bridge profile
- allowed `agentId`, `projectId`, `workspaceId`, and `clientId` lists for
  codex-memory strict policy comparison

Config values must be treated as operator-owned deployment configuration, not
as prompt-provided data.

4. Which fields must be fixed or generated by the bridge server?

The bridge server should fix or generate:

- `agentId` for the authenticated bridge runtime or bridge-side agent instance
- canonical `requestSource`, preferably `vcp-bridge`
- canonicalized `executionContext` passed to codex-memory
- any future signed-context timestamp, nonce, or signature metadata
- bridge-side session binding between authenticated caller, configured project,
  workspace, and client

The bridge server must validate any config-injected identifiers against a static
allowlist before forwarding them.

5. Which fields must never come from user prompt or tool args?

The strict authorization authority for all six fields must never come from user
prompt or public tool args:

```text
agentAlias
agentId
requestSource
projectId
workspaceId
clientId
```

This is especially strict for `agentId`, `projectId`, `workspaceId`, and
`clientId`. If those values appear in a user request or tool payload, they are
data/scope hints only and cannot authorize the write.

6. Missing `projectId` / `workspaceId` / `clientId`: how should strict fail?

In strict mode, missing required scope must fail closed before persistence. The
public rejection and audit projection must stay low-disclosure and must not echo
raw `agentId`, `workspaceId`, or `clientId` values. Observe-only behavior is
acceptable only in stage 1 rollout and must not be described as strict
enforcement.

7. Does VCP Bridge need an independent trusted context adapter?

Yes. Future integration should add a bridge-side trusted context adapter, for
example `VcpBridgeTrustedExecutionContext`, that normalizes bridge runtime
identity/scope into the same shape consumed by `RecordMemoryTrustedExecutionContext`.
The adapter should be independent from public tool arguments and should be
fixture-tested before any live bridge call.

8. Does future integration require signed context, static allowlist, or exact
approval?

Yes. Minimum future requirements:

- static allowlist for bridge identity, request source, project, workspace, and
  client
- bridge-owned context construction, not public payload parsing
- exact approval before any live bridge proof or strict production candidate
- signed context or equivalent bridge-side integrity binding before dynamic or
  session-derived identity/scope is trusted
- staged rollout: observe-only, strict temp-local, strict local candidate, then
  production candidate only after separate exact approval

## Future Contract

Future VCP Bridge write-capable integration should meet this contract before it
is considered strict-candidate-capable:

- bridge runtime supplies all six trusted fields
- bridge-side adapter canonicalizes and validates the fields
- static allowlist exists for accepted identities and scopes
- public tool payload is never used as principal/scope authority
- missing or mismatched strict fields fail closed before persistence
- public/audit rejection output remains low-disclosure
- bridge tests are fixture-only until an exact live bridge proof is approved
- no public MCP tool or schema expansion is introduced for this context
